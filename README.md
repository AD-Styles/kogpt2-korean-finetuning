## 📌 프로젝트 요약 (Project Overview)
본 프로젝트는 단순 범용 텍스트 생성 혹은 어색한 직역체 문장을 내뱉던 영문 생성에 치중된 **GPT-2** 모델의 한계를 넘어, **KoGPT2**를 기반으로 **한국어 영화 리뷰 도메인에 특화된 텍스트 생성 파이프라인**을 구축한 사례입니다. SKT의 사전 학습 한국어 모델인 **KoGPT2(`skt/kogpt2-base-v2`)** 베이스라인에 **NSMC 데이터를 파인튜닝** 하여, 기존 모델 대비 **자연스러운 구어체 표현력과 도메인 이해도**를 대폭 향상시켰습니다.

## 🎯 핵심 목표 (Motivation)
| 구분 | 세부 내용 |
| :--- | :--- |
| **성능 극대화** | 범용 지식만 가진 GPT-2 모델의 어색한 직역체를 극복하고, 영화 리뷰 분야에서 자연스러운 구어체 생성을 구사하도록 모델 최적화 |
| **도메인 특화 파인튜닝** | KoGPT2 모델에 NSMC(네이버 영화 리뷰) 데이터를 주입하여 구어체 중심의 문맥을 생성하도록 Causal LM 재학습 |
| **Tokenizer 무결성 확보** | 한국어 자연어 처리 모델에서 빈번히 발생하는 토큰 인덱스 불일치 및 한글 깨짐 문제를 디버깅하고 안정적인 인코딩 규격 확립 |

## 🛠️ 주요 알고리즘 및 기술적 구현 (Technical Implementation)

| 분류 | 세부 항목 | 구현 내용 |
| :--- | :--- | :--- |
| **Tokenization &<br>Data Preprocessing** | **정밀한 토크나이징 제어** | 일반적인 `AutoTokenizer` 대신 `PreTrainedTokenizerFast`를 명시하여 기본 토크나이저 설정이 범용 로직에 의해 변형되는 것을 차단. |
| | **Data Pipelining** | `datasets` 라이브러리를 활용, 원본 NSMC 데이터를 필터링하고 병렬로 Tokenize 처리하여 Causal LM 학습 포맷 구성. |
| **Causal LM<br>Fine-tuning Architecture** | **자원 효율적 학습 설정** | 원본 데이터를 3만 건(Train), 3천 건(Eval)으로 샘플링하고 `gradient_accumulation_steps=2`, Mixed Precision(`fp16=True`)을 적용하여 GPU 환경에서 학습 효율 극대화. |
| | **도메인 적응<br>(Domain Adaptation)** | 뉴스/위키백과 위주의 기존 가중치를 영화 리뷰 특유의 감성과 인터넷 구어체 텍스트로 재배치하여 질적 성능 차별화. |
| **Generation<br>Decoding Strategy** | **Repetition Penalty (1.2)** | 텍스트 생성 모델의 고질적인 구절 반복 문제를 제어. |
| | **Top-P (0.92)** | 확률 분포 하위 단어를 잘라내어 문맥 붕괴 예방. |
| | **Temperature (0.8)** | 안전성과 창의성 간의 밸런스를 채택하여 감성적 표현력 확보. |

## 🚀 트러블슈팅: 토크나이저 무결성 확보 (Troubleshooting)

| 구분 | 상세 내용 |
| :--- | :--- |
| **문제 현상** | **한글 인코딩 깨짐 및 `CUDA device-side assert triggered` 에러**<br>KoGPT2 모델 파인튜닝 중 지속적인 CUDA 메모리 참조 에러가 발생하였으며, 간헐적으로 출력이 외계어로 붕괴되는 현상 발생. |
| **원인 분석** | 범용 클래스인 `AutoTokenizer`가 KoGPT2를 초기화할 때, 기본 어휘 사전 크기(`51,200`)를 넘어 패딩을 위한 새로운 토큰을 암시적으로 추가하여 임베딩 크기를 `51,201`로 변형시킴.<br>이로 인해 모델 학습 시 전체 인덱스 맵핑이 1칸씩 밀리면서, 올바른 벡터가 출력되어도 잘못된 글자로 강제 디코드되는 현상이 발생. |
| **해결 방안** | 파생 토큰 생성을 방지하기 위해 `PreTrainedTokenizerFast`를 명시적으로 선언하고, `bos_token`, `eos_token`, `pad_token`을 본래 KoGPT2 체계에 맞게 하드코딩.<br>불필요한 `model.resize_token_embeddings()` 과정을 해제하여 원래 모델 구조(Vocab Size 51,200)에 완벽히 종속되도록 코드를 수정함으로써 무결점의 한국어 출력 확보. |

## ⚠️ 한계점 및 개선 방향 (Limitations & Future Work)

성공적인 도메인 적응(Domain Adaptation)을 이루어냈으나, 생성된 추론 결과를 바탕으로 모델의 한계점을 명확히 인지하고 다음과 같은 고도화 방향성을 도출했습니다.

| 분류 | 한계점 (Limitations) | 개선 방향 (Action Plans) |
| :--- | :--- | :--- |
| **문맥 일관성** | **감정선의 혼선 (Incoherent Sentiment)**<br>하나의 리뷰 내에서 칭찬과 비판이 교차하는 등 일관된 태도(Stance)를 유지하지 못하는 현상이 발생함. | **Conditioned Generation 도입**<br>데이터셋의 긍정/부정 라벨을 프롬프트에 활용하여(예: `[긍정] 배우들의 연기가`), 특정 감정선에 고정된 텍스트를 생성하도록 추가 학습 로직 구현. |
| **데이터 노이즈** | **특수기호 과적합 (Symbol Overfitting)**<br>정제되지 않은 날것의 데이터가 주입되어, 과도한 이모티콘이나 의미 없는 기호의 연속(`***-_***;;;`)이 생성됨. | **정밀한 전처리 파이프라인 구축**<br>정규표현식(Regex)을 고도화하여 무의미한 자음/모음 반복 및 특수문자 조합을 사전에 필터링하는 데이터 정제 강화. |
| **문장 생성 제어** | **어색한 문장 종료 (Abrupt Ending)**<br>`max_length` 한계에 도달하거나 EOS 토큰이 적절히 발동하지 않아 문장이 중간에 잘리는 현상 존재. | **디코딩 파라미터 및 후처리 로직 조정**<br>`early_stopping=True`를 활용하거나, 생성 완료 후 마지막 구두점(., !, ?)을 기준으로 텍스트를 깔끔하게 자르도록 파이프라인 후처리 추가. |

## 💡 회고 (Retrospective)
이 실습을 통해 언어 모델의 근간이 되는 '토크나이저'가 얼마나 민감한 부품인지 체감할 수 있었습니다. 특히, 단순히 라이브러리의 경고 메시지를 숨기거나 API를 끌어다 쓰는 수준을 넘어 **내부의 행렬 크기(Vocabulary Size) 통제 구조와 Tokenizer 인덱싱 과정**을 깊숙이 파헤치고, 트러블슈팅의 핵심 원리를 관통해낸 경험은 일반적인 튜토리얼에서는 결코 얻을 수 없는 인사이트였습니다.
단순한 텍스트 훈련을 넘어서, 인공지능이 데이터를 다루는 '바이트와 인덱스 로직' 자체의 지배력을 기르게 된 뜻깊은 포트폴리오로 자리매김했습니다.

  무엇보다 가장 큰 성과는 **모델의 성능 변화를 시각적, 정성적으로 바로 체감할 수 있었다**는 점입니다. 단순 영문을 나열하거나 엉뚱한 뉴스 문맥을 읊던 Base 언어 모델이, 파인튜닝을 거친 후 영화에 대한 인간의 감정을 그대로 모방한 한국어 문장을 술술 뱉어내는 것을 보며 **'양질의 도메인 특화 데이터가 텍스트 생성 품질에 미치는 압도적 폭발력'**을 증명해낼 수 있었습니다. 

단순히 API를 호출하는 것을 넘어, 언어 모델의 근간이 되는 토크나이저의 내부 행렬 크기(Vocabulary Size) 통제와 인덱싱 과정을 깊이 있게 이해할 수 있는 실습이었습니다. 트러블슈팅 과정을 통해 데이터의 '바이트와 인덱스 로직'을 제어하는 방법을 습득했으며, 양질의 도메인 특화 데이터가 텍스트 생성 품질에 미치는 긍정적인 영향을 시각적으로 명확히 확인할 수 있었습니다.
